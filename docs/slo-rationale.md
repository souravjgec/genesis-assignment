The genesis-items-api has an SLO of 99.0% availability in 30 days because it is an early-stage Lambda-based internal API with a simple deployment model: a public Function URL.  Aiming for a target of 99.9% would mean only around 43.2 minutes of downtime in 30 days. This is a bit too aggressive. After all, the severity level of deployment flow, observability, alert is still small. In contrast, 432 minutes of downtime are allowed in a 30-day period for 99.0, which creates a practical error budget while still holding the service to a meaningful reliability standard.

Depending on the current architecture and data that are already accessible, not on the aspirational end state, was this target. The Lambda Errors and Invocations metrics are already published to CloudWatch by the service, and an alarm for high error rate and a dashboard are already created so we can directly measure availability using AWS-native telemetry. Similarly, currently there is not a full month of historical traffic, incident history or burn-rate data for a more aggressive commitment. Considering the maturity of the service and expected internal usage as well as the fact that it is not yet a hardened multi-tier production system, 99.0% is a good starting point.

Once I have a month’s worth of real traffic, I will look at error-rate trends, deployment failure patterns, alarm frequency, and actual error-budget burn. If the service remains consistently healthy, I could think about tightening the target or setting independent SLOs for latency and availability.
